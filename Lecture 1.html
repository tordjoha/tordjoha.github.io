First Lecture - Software security

Motivation

  • bad security has economic and privacy consequences, and security bugs impacts user experience.
  • Secure code is often good code. Easy to reason about - easier to maintain - has less (other) bugs.
  • Security is not a feature - has to be assigned for the first second you think about a program.
  • How to produce secure code: understanding - the underlying systems, the problem domain and what the pitfalls and footguns are.

Security is not a feature

Example: A login page can be correct, but still fail to secure the website.

Programs

  • Executable file: programs can be created from source code in different ways.
  • Process: the operating system (OS) manages the different processes running. Each process has:
    • a process ID (which identifies the process)
    • a (virtual) memory space
    • a lot of other metadata(UID, PID, file descriptors,...)
    • How the memory space is managed is called the /runtime environment/, and depends on the computer
    • In particular, the OS does not know about:
    • Variables
    • types
    • objects
    • scopes
    • functions

Runtime environment

Difference between C and Java

  • C - very thin abstraction over the OS model of a process, Java provides a more robust abstraction. Java guarantee: memory safety

Processes interacting with the environment

Processes make system calls to interact with the world beyond its memory space

  • read / write to files
  • network access
  • time functions
  • inter-process communication

The OS receives the call and executes the action on behalf of the process. The system call interface is a security boundary
